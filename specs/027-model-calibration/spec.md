# Feature Specification: Model Calibration

**Feature Branch**: `027-model-calibration`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "The Lead Quant Mandate - Model Calibration... Monitor the gRPC latency between the Python Orchestrator and the Java Virtual Threads. Is it sub-millisecond? Analyze the simulated fills from Shadow Mode. Are our target prices realistically achievable given the top-of-book liquidity? Check the Redis idempotency locks. Under load, are we double-firing execution signals?"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Latency Performance Audit (Priority: P1)

As a quant engineer, I want to monitor the end-to-end gRPC communication delay between the Python Orchestrator and the Java Execution Engine, so I can ensure our arbitrage signals are not becoming stale before they hit the market depth layer.

**Why this priority**: Latency is the primary killer of high-frequency arbitrage. Sub-millisecond performance is a prerequisite for live deployment.

**Independent Test**: Can be tested by running a continuous execution loop in Shadow Mode and logging the "Orchestrator-to-Engine" RTT (Round Trip Time) in both services.

**Acceptance Scenarios**:

1. **Given** a high-frequency signal burst, **When** the Python Orchestrator sends a trade request, **Then** the Java Engine must receive it within 500μs.
2. **Given** continuous operation, **When** the total latency (including gRPC and Engine processing) exceeds 2ms, **Then** the system must generate a performance warning.

---

### User Story 2 - Fill Achievability Analysis (Priority: P1)

As a quantitative analyst, I want to audit the simulated fill prices generated by Shadow Mode against the available L2 liquidity at the time of "execution," so I can confirm that our arbitrage spreads are actually capturable in real market conditions.

**Why this priority**: Theoretical alpha must match achievable execution. If we can't fill at the Z-score target, the strategy is unviable.

**Independent Test**: Verify that for every Shadow Trade, the `reasoning_metadata` contains the L2 snapshot and the VWAP calculation is within the spread.

**Acceptance Scenarios**:

1. **Given** a Shadow Trade, **When** the quantity exceeds the top level of the L2 book, **Then** the recorded price must accurately reflect the "walk-the-book" VWAP, not just the mid-price.

---

### User Story 3 - Idempotency Stress Test (Priority: P2)

As a system operator, I want to verify that our Redis-based idempotency locks are robust under heavy load, so we never double-fire execution signals to the broker during a market spike.

**Why this priority**: Double-firing orders would lead to catastrophic loss and risk exposure.

**Independent Test**: Use a load generator to send 100 duplicate `signal_id` requests per second and verify zero duplicate executions.

**Acceptance Scenarios**:

1. **Given** multiple identical signal requests arrive simultaneously, **When** the first one is being processed, **Then** all subsequent requests must receive a "Duplicate Request" status and not touch the broker/simulation logic.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST implement a high-precision latency logging interceptor in both gRPC client (Python) and server (Java).
- **FR-002**: System MUST calculate the "Alpha Stale Time" as (Engine Received Time - Orchestrator Sent Time).
- **FR-003**: System MUST provide a report or log of Shadow Mode fill accuracy vs. theoretical mid-price.
- **FR-004**: System MUST verify L2 snapshot validity for every trade to ensure fill price realism.
- **FR-005**: Redis idempotency check MUST use atomic `SET NX` or Lua scripting to ensure thread safety under concurrent load.
- **FR-006**: System MUST generate a `LATENCY_ALARM` if gRPC RTT consistently exceeds 1ms.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Average gRPC latency between Python and Java is under 0.8ms.
- **SC-002**: Shadow Mode fill prices are 100% verifiable against the captured L2 snapshot.
- **SC-003**: Zero (0) duplicate execution records in the `TradeLedgerRepository` when processing identical signals under load.
- **SC-004**: System successfully identifies 100% of "unachievable" targets where spread < simulated slippage.

## Assumptions

- **Clock Synchronization**: Python and Java environments have synchronized system clocks for accurate latency measurement.
- **L2 Snapshots**: Shadow Mode is active and capturing L2 data as per feature 026.
- **Redis Availability**: High-speed Redis instance is available for the idempotency locking.
- **Load Generation**: Tools are available to simulate high-frequency signal bursts.
